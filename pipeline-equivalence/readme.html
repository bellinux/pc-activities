Guía de Análisis de Equivalencia (Microbit vs Protobject)

Este documento explica cómo se agrega una actividad al sistema y se genera todo el análisis comparativo: desde la extracción del código hasta los grafos y las métricas algorítmicas.

El pipeline tiene tres tipos de paso:

  • DETERMINÍSTICO Extracción del código a texto — ahora la hace un decodificador en JavaScript (Node), directamente desde los archivos fuente .ptj / .hex. Reproducible, sin intervención humana.
  • LLM Abstracción semántica (AST → Tabla IR canónica → PDG) — la realiza el modelo (agente), replicando el esquema del ejemplo vela.
  • DETERMINÍSTICO Cálculo matemático (grafos SVG + métricas) — script de Python con NetworkX y Graphviz.
Cambio importante: la extracción YA NO usa imágenes (microbit.png / protobject.png) ni un LLM que las "lea". El texto se extrae de forma determinística y reproducible de los .ptj/.hex con blocks-decoder.js — el mismo decodificador que usa el visor "Ver bloques en vivo". Si cambias una fuente, basta regenerar.

🤖 AGENTE: estudia el ejemplo antes de abstraer

Para los pasos LLM (AST, PDG, Tabla IR) es obligatorio replicar exactamente el esquema y los nombres de llaves del ejemplo en reporte/actividades/vela/:

  • ast_makecode.json / ast_protobject.json — árbol jerárquico con children.
  • tabla_ir.json — ontología canónica: mapeo canonical_node ↔ makecode_block ↔ protobject_block. Define el vocabulario.
  • pdg_makecode.json / pdg_protobject.json — nodes (id, kind, seq, in_loop) y edges (source, target, type ∈ control/data/side_effect, var). El kind de cada nodo ES el canonical_node de la Tabla IR.

Regla clave de la abstracción: a operaciones semánticamente equivalentes en ambas plataformas se les asigna el mismo kind canónico. Pero se representa cada programa tal cual es: si difieren, los grafos difieren (y la métrica lo refleja). No se fuerza la equivalencia.

Flujo de Trabajo

Paso 1: La actividad ya existe (sin imágenes) DETERMINÍSTICO

Las 17 actividades ya están definidas en pipeline-equivalence/actividades.json (orden, code, título). Sus fuentes están en la raíz del proyecto:

  • protobject/equivalent-<code>.ptj
  • microbit/<code>.hex

El nombre de la carpeta de cada actividad en reporte/actividades/ es su code (ej. juego-toques).

Paso 2: Extracción de texto DETERMINÍSTICO (Node)

Genera text_makecode.json y text_protobject.json para las 17 actividades con un solo comando (desde la raíz del proyecto):

node gen-text-json.js

Para una sola actividad puedes usar el decodificador directamente:

node decode.js protobject/equivalent-juego-toques.ptj node decode.js microbit/juego-toques.hex
El JSON tiene un arreglo de strings bajo "extracted_text" (modo "solo estructura": sin comentarios, LEDs colapsados), preservando la indentación con espacios.

Paso 3: AST — Árbol de Sintaxis Abstracta LLM

A partir de los textos extraídos, genera la estructura jerárquica de cada plataforma (todavía con etiquetas propias de cada una):

  1. Crea ast_makecode.json y ast_protobject.json con anidamiento children (ver vela).

Paso 4: Tabla Ontológica IR — vocabulario canónico LLM

Antes del grafo hay que fijar el vocabulario canónico: a qué concepto común (canonical_node) corresponde cada bloque de cada plataforma. A operaciones semánticamente equivalentes se les asigna el mismo canonical_node.

  1. Crea tabla_ir.json: arreglo de objetos canonical_node / makecode_block / protobject_block.
Este es el puente semántico entre las dos plataformas: define los nombres canónicos que usará el PDG. Por eso va antes del grafo.

Paso 5: PDG — Grafo de Dependencia (sobre el IR canónico) LLM

Construye el grafo de dependencias usando el vocabulario de la Tabla IR: el kind de cada nodo es el canonical_node correspondiente. Las aristas son las dependencias de control, de datos y de efecto colateral. Así, dos programas equivalentes producen grafos isomorfos.

  1. Crea pdg_makecode.json y pdg_protobject.json (nodes con kind canónico + edges control/data/side_effect).

Paso 6: Pipeline Algorítmico DETERMINÍSTICO (Python)

Con todos los JSON generados, el cálculo es automático:

cd pipeline-equivalence/reporte/scripts/ python run_pipeline.py juego-toques

Lee pdg_makecode.json y pdg_protobject.json, construye los grafos con NetworkX, exporta pdg_makecode.svg / pdg_protobject.svg con Graphviz, calcula Graph Edit Distance e Isomorfismo (VF2) y guarda metricas.json.

Requisito: el ejecutable Graphviz dot debe estar en el PATH (para el render SVG). Las métricas también se guardan aunque falte.

Paso 7: Dashboard AUTOMÁTICO

El dashboard index.html es ahora dinámico: lee actividades.json y construye solo la lista ordenada de las 17 actividades. Ya NO se copia/pega HTML. Si reordenas o agregas actividades, regenera el índice con node gen-actividades-json.js. Mientras una actividad no tenga metricas.json, se muestra como "Pendiente" (la extracción de texto ya funciona). vela queda como ejemplo hardcodeado de referencia.

Paso 8: Visualización y verificación

  1. Levanta el servidor desde pipeline-equivalence/:
    2. python -m http.server 8080
  2. Abre http://localhost:8080 y revisa los visores (Extracción, AST, Tabla IR, PDG) de cada actividad.
